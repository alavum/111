# RUBEZH — Инструкция по запуску и деплою (на русском)

Этот репозиторий содержит фронтенд (Vite + React/Tailwind) и простой бэкенд (Express) для API. Контент (новости, правила, политика, офферта и пр.) хранится в виде файлов в каталоге `data/` (JSON). Загрузка изображений сохраняется в папку `uploads/`.

Внимание: текущая реализация сохраняет данные в файловой системе (data/*.json и uploads/). На бессерверных платформах (Netlify Functions, Vercel Functions и т.п.) записи в локальную файловую систему не стабильны и НЕ гарантируют сохранность данных между вызовами. Для продакшена рекомендуется подключить БД или внешнее хранилище (см. раздел «Рекомендации»).

Требования
- Node.js 18+ (рекомендуется LTS)
- npm, yarn или pnpm (в репозитории указан pnpm в packageManager, но npm/pnpm/yarn подойдут)
- Резервное копирование каталога `data/` и `uploads/`, если вы используете файловое хранилище

Быстрый старт (локально)
1. Склонируйте репозиторий и перейдите в папку проекта.
2. Установите зависимости:
   - pnpm: `pnpm install`
   - npm: `npm install`
   - yarn: `yarn install`
3. Запустить в режиме разработки (включает Vite dev сервер):
   - `npm run dev` (или `pnpm run dev` / `yarn dev`)
4. Откройте браузер: http://localhost:5173 (порт указан в Vite; в конфиге dev порт 8080, но Vite по умолчанию использует 5173 — проверьте вывод команды)

Сборка и запуск в продакшене (без Docker)
1. Соберите клиент и сервер:
   - `npm run build`
   Это выполняет `npm run build:client` (vite build) и `npm run build:server`.
2. Запустите собранный сервер:
   - `npm start`
   Команда запускает `node dist/server/node-build.mjs` (в зависимости от сборки). Сервер будет обслуживать API и статическую папку `dist/spa`.

Локальный продакшен-сервер — шаги:
- Убедитесь, что папки `data/` и `uploads/` доступны и имеют права записи.
- Подготовьте файл `.env` (см. раздел ниже). Затем выполнить `npm run build` и `npm start`.

Размещение на Netlify (с функциями)
- В проекте уже есть `netlify.toml` (настроен: build command = `npm run build:client`, functions = `netlify/functions`, publish = `dist/spa`).
- В Netlify при подключении репозитория укажите:
  - Build command: `npm run build:client`
  - Publish directory: `dist/spa`
  - Functions directory: `netlify/functions`
- Важно: Netlify Functions — бессерверные. Любые изменения файловой системы (запись в `data/*.json` или `uploads/`) не будут храниться надёжно. Если вам нужна сохранность данных, используйте удалённую БД или CMS.
- Подключение через Builder.io MCP: если хотите использовать автоматический деплой через MCP, подключите Netlify в разделе MCP ([Connect Netlify](#open-mcp-popover)).

Настройки окружения (.env)
- Проект использует dotenv для серверной части (server/index.ts импортирует `dotenv/config`). Создайте файл `.env` в корне проекта (не коммитьте в VCS).
- Рекомендуемые переменные:
  - JWT_SECRET="введите_сложную_строку_секрета"
  - ADMIN_PASSWORD="менее_предпочтительно_в_коде_—_лучше_в_переменных"
  - PORT=3000 (порт, если запускать Node сервер отдельно)
  - PING_MESSAGE="pong" (необязательно)
  - RCON_1_IP, RCON_1_PORT, RCON_1_PASSWORD — если вы решите вынести конфиг RCON в env
- Пример `.env`:
  JWT_SECRET="replace_this_with_a_strong_secret"
  ADMIN_PASSWORD="very_strong_admin_password"
  PORT=3000

Безопасность
- Никогда не храните реальные пароли и секреты в публичном хранилище кода. Используйте переменные окружения и секреты провайдера хостинга.
- Если вы используете RCON пароли или другие критические значения, храните их в безопасном секретном менеджере (например, в окружении VPS, в секретах Netlify/Vercel, или в MCP-интеграциях).

Персистентность данных — важное замечание
- Текущая реализация записывает в `data/*.json` и `uploads/`.
- На сервере с постоянным диском (VPS, выделенный сервер, Docker-контейнер с примонтированным volume) это работает нормально.
- На бессерверных платформах (Netlify/Vercel Functions) запись в локальную файловую систему НЕ является перманентной. Для продакшена рекомендуем:
  - Перенести хранение контента в базу данных (Neon, Supabase, Postgres и т.д.) или
  - Использовать Builder CMS (подключается через MCP) для редактирования и хостинга контента или
  - Хранить файлы/изображения в объектном хранилище (S3/Backblaze/Cloudflare R2) и сохранять метаданные в БД.

FAQ / Частые действия при подготовке к проду
- Как заменить публичные ссылки (Discord/YouTube и т.д.)? Откройте `client/components/Footer.tsx` и `client/components/Hero.tsx`, отредактируйте `href` в соответствующих массивах/компонентах.
- Где редактировать тексты правил/политики/новостей? Файлы в `data/` — `rules.json`, `privacy.json`, `terms.json`, `news-articles.json`. Их можно редактировать вручную или через API (`/api/news`, `/api/rules` и т.д.).
- Как сделать хранение постоянным при развертывании на Netlify? Перенесите storage из файловой системы в облачную БД (Supabase/Neon) или используйте Builder CMS.

Если что-то не получается
- Для деплоя через Netlify/Vercel используйте их GUI и укажите указанные выше команды и директории.
- Для хостинга на VPS: скопируйте проект на сервер, установите зависимости, создайте `.env`, запустите `npm run build` и `npm start`.
